# Startup Security Baseline Policy

**Effective Date:** January 2026  
**Policy Owner:** Security & Compliance Lead  
**Target Audience:** All startup employees, contractors, and founders  
**Classification:** Internal Use

---

## Executive Summary

This Startup Security Baseline Policy defines the minimum security controls required to protect company data, assets, and reputation. It is designed for early-stage startups to implement practical, high-impact security measures with limited resources, supporting compliance and customer trust from day one.

---

## 1. Policy Purpose & Scope

### 1.1 Purpose

- Establish a security foundation for all company operations
- Protect sensitive data, intellectual property, and customer information
- Meet basic regulatory and contractual security requirements
- Enable secure growth and scalability

### 1.2 Scope

This policy applies to:

- All employees, founders, and contractors
- All company-owned and personal devices used for work
- All cloud services, SaaS platforms, and internal systems

---

## 2. Core Security Controls

### 2.1 Access Management

- Use unique accounts for every user (no shared logins)
- Enforce strong passwords (12+ characters, no reuse)
- Enable Multi-Factor Authentication (MFA) on all critical systems (email, cloud, code repos)
- Remove access immediately upon role change or departure

### 2.2 Device Security

- Require full-disk encryption on all laptops/desktops
- Keep operating systems and software up to date (auto-updates enabled)
- Install and enable antivirus/anti-malware protection
- Enable device auto-lock after 10 minutes of inactivity

### 2.3 Data Protection

- Store sensitive data only in approved, secure locations (no local storage for customer data)
- Use encryption for data at rest and in transit (TLS/SSL for all web apps)
- Back up critical data at least weekly; test restores quarterly

### 2.4 Cloud & SaaS Security

- Use reputable, secure SaaS providers (SOC 2, ISO 27001 preferred)
- Restrict admin privileges to minimum required
- Review and remove unused accounts/services quarterly

### 2.5 Secure Development

- Use version control (e.g., GitHub, GitLab) for all code
- Require code review for all production changes
- Scan code for vulnerabilities before deployment (use free tools like GitHub Dependabot)

### 2.6 Incident Response

- Report security incidents immediately to the Security Lead
- Document and investigate all incidents
- Change passwords and review access after any breach

---

## 3. Roles & Responsibilities

| Role                | Responsibility                                 |
|---------------------|------------------------------------------------|
| Security Lead       | Policy owner, incident response, training       |
| Founders/Executives | Enforce policy, allocate resources              |
| Employees           | Follow policy, report issues, protect data      |
| Contractors         | Comply with policy, use secure practices        |

---

## 4. Training & Awareness

- Complete basic security awareness training within 30 days of joining
- Annual refresher training for all staff
- Phishing and social engineering awareness included

---

## 5. Review & Maintenance

- Policy reviewed annually or after major incident
- Update controls as company grows or threat landscape changes

---

## 6. Quick Reference Checklist

- [ ] MFA enabled on all accounts
- [ ] Laptops encrypted and auto-lock enabled
- [ ] Antivirus/anti-malware installed
- [ ] Strong, unique passwords in use
- [ ] Sensitive data only in approved locations
- [ ] Code in version control, reviewed before deploy
- [ ] Backups scheduled and tested
- [ ] All staff trained on security basics
